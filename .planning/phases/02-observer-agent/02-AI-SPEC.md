## 1. System Classification

**System Type:** Autonomous Agent (Learning Extraction + Classification)

**Description:**
The @observer is a Claude Code subagent that reads structured event logs (obs.jsonl) from completed agent runs, extracts reusable learnings, classifies each learning to the correct memory tier via 3-layer scope-test questions, and writes tagged entries to Pending Review sections. "Good" means: high-signal insights that actually improve future agent runs, classified to the right scope, with no noise (duplicates, trivial observations, or self-referential loops).

**Critical Failure Modes:**
1. **Self-loop amplification** — observer observes its own runs, creating recursive noise that fills memory files with meta-observations about observing
2. **Scope misclassification** — insight written to wrong memory tier (skill insight → agent memory, or vice versa), causing wrong agents to act on it
3. **Noise flooding** — low-quality or trivial observations pass through, degrading signal-to-noise ratio in memory files until they become useless
4. **Hallucinated evidence** — observer fabricates or misattributes learnings not supported by the actual obs.jsonl events
5. **Duplication accumulation** — same insight proposed repeatedly across invocations, filling Pending Review with redundant entries

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer Tooling / Knowledge Management

**User Population:** Solo developer who reviews proposed learnings via /evolve command; the observer itself has no end-user interaction — its output is staged for human review before promotion to permanent memory.

**Stakes Level:** Medium

**Output Consequence:** Accepted insights become permanent memory that all future agent runs read and follow. Bad insights compound silently — agents act on flawed guidance, producing subtly degraded work across all downstream runs. Good insights compound positively — agents avoid repeating mistakes and adopt proven strategies without human re-intervention.

### What Domain Experts Evaluate Against

```
Dimension: Actionability
Good: Insight prescribes a specific behavioral change an agent can follow —
      "Always check file existence before Read calls to avoid retry loops"
Bad: Insight states an obvious fact or vague observation —
     "The researcher agent did research" or "errors should be handled properly"
Stakes: Critical
Source: Knowledge management practitioner consensus — lessons learned systems
        fail when they accumulate platitudes instead of operational guidance
```

```
Dimension: Evidence grounding
Good: Insight traces directly to a specific event in obs.jsonl — names the
      agent, the tool call, the outcome, and why it matters
Bad: Insight states a plausible-sounding principle that cannot be verified
     against the actual run data — sounds true but may be fabricated
Stakes: Critical
Source: Self-improving agent research (Nakajima 2024-2025) — "if the model
        hallucinate bad reflections and reinforce them," the knowledge base
        becomes actively harmful over time
```

```
Dimension: Scope precision
Good: Insight is written to the exact memory tier where it will be consumed —
      a tool-usage pattern goes to the skill's insights.md, an agent behavioral
      pattern goes to that agent's MEMORY.md, a cross-agent coordination issue
      goes to PLAYBOOK.md
Bad: Insight is written to a plausible-but-wrong tier — e.g., a researcher-
     specific workflow tip placed in a shared skill file where all agents read
     it and incorrectly apply it to their own contexts
Stakes: High
Source: Taxonomy quality research (Unterkalmsteiner et al. 2023) — classification
        systems fail when categories overlap or items are placed in "close enough"
        locations rather than the precisely correct one
```

```
Dimension: Non-redundancy
Good: Insight adds genuinely new knowledge not already present in the target
      file — either a new pattern, a refinement of an existing insight, or a
      contradiction that supersedes old guidance
Bad: Insight restates something already written in the target file, possibly
     with slightly different wording — reviewer wastes time comparing near-
     duplicates
Stakes: Medium
Source: Knowledge base maintenance practice — duplication is the primary cause
        of knowledge base bloat that eventually makes the entire system unusable
        (knowledge management cycle literature)
```

```
Dimension: Confidence calibration
Good: HIGH confidence tag applied only when the evidence is unambiguous (clear
      error-recovery sequence, explicit tool failure with workaround); LOW
      applied when the pattern was observed once and may be coincidental
Bad: Everything tagged HIGH regardless of evidence strength, or everything
     tagged LOW defensively — both destroy the signal value of the tag for
     the reviewer doing accept/reject decisions
Stakes: Medium
Source: Practitioner knowledge — categorical confidence must track actual
        evidence strength to enable fast reviewer triage; miscalibrated
        confidence forces the reviewer to re-evaluate every entry from scratch
```

### Known Failure Modes in This Domain

1. **Knowledge base poisoning via hallucinated reflections** — The observer fabricates a plausible-sounding lesson that was never actually demonstrated in the run. Once promoted to permanent memory, all future runs inherit flawed guidance. This is the self-improving agent equivalent of training on corrupted data. Research on self-evolving agents identifies this as the primary risk: "if the model can hallucinate bad reflections and reinforce them," performance degrades over time.

2. **Comfort-zone extraction (curriculum collapse)** — The observer repeatedly extracts the same type of low-risk, obvious insight (e.g., "file paths should be correct") while missing genuinely novel patterns in the data. This mirrors the documented "curriculum collapse" phenomenon where self-improving systems stay near their comfort zone unless explicitly pushed toward diversity.

3. **Scope drift through progressive misclassification** — An insight that belongs to one agent's memory gradually migrates to broader scope through repeated observer runs. For example: a researcher-specific citation pattern gets classified as a skill-level insight, then all agents start applying it inappropriately. Taxonomy research identifies this as "boundary erosion" — categories that seem clear in isolation become ambiguous under repeated classification pressure.

4. **Stale knowledge persistence** — The observer extracts a valid insight from an early run, but the underlying system changes (tool updated, agent prompt rewritten, workflow restructured). The insight remains in memory and agents follow now-outdated guidance. Research on experience libraries warns that "if the environment or objectives shift, old trajectories can become misleading."

## 2. Framework Decision

**Selected Framework:** Claude Code Native Subagent (no external AI SDK)

**Version:** Claude Code harness (current) + Sonnet 4.6 model

**Rationale:**
The observer operates entirely within the Claude Code runtime as a dispatched subagent (`.claude/agents/observer.md`). The project's billing constraint requires all LLM calls route through Claude Code subagent dispatches (Claude Max subscription). The observer's task — reading JSONL, extracting patterns, classifying to memory tiers — is a prompt-engineering problem, not a framework orchestration problem. No external SDK, vector store, or agent framework is needed. The harness provides: tool access (Read, Write, Grep, Glob, Bash), agent definition format (YAML frontmatter + markdown body), and dispatch infrastructure.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Claude Agent SDK (Python) | Would require direct API calls billed outside Claude Max; adds dependency for no capability gain over native subagent |
| LangGraph | Stateful graph orchestration is overkill for a single-pass read→classify→write pipeline |
| CrewAI | Multi-agent orchestration unnecessary — observer is a single agent, not a crew |
| Custom Python script | Loses LLM classification capability; would need API key (billing violation) |

**Vendor Lock-In Accepted:** Yes — intentional. Observer runs within Claude Code, which is already the project's runtime. No portability requirement exists.

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| Agent Definition (.md) | YAML frontmatter (model, tools, skills) + markdown system prompt in `.claude/agents/` | Defines the observer's identity, capabilities, and behavioral contract |
| Subagent Dispatch | Parent conversation invokes `@agent-name <prompt>` to spawn a child agent | /evolve command triggers the observer; observer itself never dispatches children |
| Tool Access List | `tools:` array in frontmatter restricts which Claude Code tools the agent can use | Observer needs Read (obs.jsonl), Write/Edit (memory files), Grep (dedup), Glob (find targets), Bash (cursor ops) |
| Skills Injection | `skills:` array in frontmatter auto-injects SKILL.md content into agent context at startup | Observer inherits agent-protocols for memory lifecycle conventions |
| Hook Events | JS scripts registered in settings.json fire on PreToolUse, PostToolUse, SubagentStop, etc. | pipeline-observe.js captures the events that become observer's input; SubagentStop fires when observer completes (observer's own events are captured but self-filtered) |

### Common Pitfalls

1. **Self-observation loop** — `pipeline-observe.js` captures ALL tool events including the observer's own Read/Write/Edit calls. If the observer processes its own events on the next invocation, it generates meta-observations about observing. Prevention: the observer's prompt must filter events where `agent_id === "observer"` and skip them entirely. The hook does NOT filter (it captures everything for audit trail completeness).

2. **Context window exhaustion on large obs.jsonl** — A 10MB obs.jsonl at rotation threshold contains ~5,000-10,000 events. Reading the entire file in one Read call consumes most of Sonnet's 200K context window. The Read tool's 2000-line default limit means only ~2000 events load per call anyway, but even that may overflow context when thinking blocks are included (10KB per turn). Mitigation: cursor-based incremental reads; process one run at a time; stop and update cursor after 8 runs.

3. **Write conflicts on memory files** — If the observer runs concurrently with another agent writing to the same MEMORY.md (possible if user dispatches agents manually while /evolve is running), the Edit tool may produce merge artifacts or lose data. Mitigation: observer only runs via /evolve dispatch (no concurrent execution by design); /evolve is a user-initiated command that implies no other agents are running.

4. **Markdown formatting corruption** — Writing entries with incorrect heading levels, missing blank lines, or malformed list items corrupts the 5-section MEMORY.md structure. Since MEMORY.md is auto-injected (first 200 lines) into every agent's context at task start, corruption propagates to all agents. Mitigation: observer reads the target file first, identifies the exact `## Pending Review` section boundaries, and appends after the last existing entry with correct formatting.

5. **Cursor drift after file rotation** — When obs.jsonl rotates (hits 10MB cap), the file is renamed to `obs.archive/obs-<timestamp>.jsonl` and a new empty obs.jsonl is created. The cursor's byte offset now points into a non-existent position. If the observer resumes with a stale cursor, it either reads nothing (offset > file size) or skips unprocessed events from the old file. Mitigation: cursor stores both byte offset AND the `epoch_ms` of the last processed event; on startup, if byte offset exceeds file size, scan from line 0 and skip events with `epoch_ms <= cursor.last_epoch_ms`.

## 4. Implementation Guidance

**Model Configuration:**

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Model | `sonnet` (Claude Sonnet 4.6) | Balanced reasoning for extraction/classification; 200K context in subagent mode handles batched runs; covered by Claude Max subscription |
| Temperature | Not configurable | Claude Code subagent dispatch does not expose temperature. Output consistency controlled via prompt engineering (explicit format, few-shot examples, enumerated steps) |
| Max tokens | Not configurable | Harness-managed. Observer output is bounded by design: 0-3 entries per run, each 1-2 sentences. No risk of runaway generation |
| Extended thinking | Not available | Subagent dispatch does not enable extended thinking. Compensated by explicit chain-of-thought instructions in the system prompt ("Before classifying, state which scope-test question each candidate answers") |

**Core Pattern: Read - Batch - Extract - Classify - Write**

```
PIPELINE (single observer invocation):

1. RESUME
   Read .observer-cursor → {byte_offset, last_epoch_ms, last_run_id}
   If missing → first invocation, start from byte 0
   If obs.jsonl missing/empty → exit "no events to process"

2. LOAD EVENTS (incremental)
   Bash: tail -c +<byte_offset> .claude/logs/observations/<project>/obs.jsonl
   Parse JSONL lines into events array
   Skip events where agent_id === "observer"  [OBSV-06: self-loop prevention]

3. GROUP INTO RUNS
   A run = sequence of events sharing same agent_id, bracketed by
   event type "dispatch" (start) and "complete" (end)
   Sort runs by epoch_ms ascending (process oldest first)

4. PER-RUN EXTRACTION (0-3 candidates max)
   For each run:
     Read thinking blocks (richest signal source: WHY decisions were made)
     Read tool failures + recoveries (error pattern detection)
     Read successful strategies (what worked well)
     Identify tool usage anti-patterns (slow, redundant, failed approaches)
     Note coordination issues (if multi-agent handoff visible)
     Extract 0-3 candidate learnings
     Each candidate: 1-2 sentence distilled insight

5. CLASSIFICATION (per candidate, scope-test enforcement) [OBSV-03]
   Evaluate THREE scope-test questions:
     Q1: "Does this change how the skill/method runs?"
         YES → target: .claude/skills/<skill>/insights.md
     Q2: "Would a fresh instance of this agent need this to do its job?"
         YES → target: .claude/agent-memory/<agent>/MEMORY.md
     Q3: "Does this change how agents hand off or coordinate?"
         YES → target: .claude/PLAYBOOK.md
   Rule: EXACTLY ONE question must pass.
     Zero pass → reject (reason: "no-scope-match")
     Multiple pass → reject (reason: "ambiguous-scope")

6. CONFIDENCE ASSIGNMENT [MEML-01]
   [HIGH] — Unambiguous evidence: clear error-recovery, explicit tool failure
            with documented workaround, repeated pattern across multiple turns
   [MED]  — Likely useful but single-instance evidence: observed once, agent
            reasoning suggests it will recur but not yet proven
   [LOW]  — Speculative: plausible pattern from limited data, or inference
            from thinking blocks without observable tool-level confirmation

7. DEDUPLICATION [OBSV-05]
   For each surviving candidate:
     Grep target file for key phrases from the candidate
     Read matching section context (5 lines around matches)
     LLM judgment: "Is this substantially the same insight as an existing entry?"
     If yes → reject (reason: "duplicate-of-existing")

8. WRITE ENTRIES [OBSV-04]
   Format for MEMORY.md / PLAYBOOK.md:
     - [CONFIDENCE] source-agent: distilled insight (YYYY-MM-DDThh:mm evidence-pointer)
   Format for insights.md:
     - [YYYY-MM-DD] [CONFIDENCE] distilled insight (from: source-agent, evidence-pointer)
   Use Edit tool: target the ## Pending Review section, append after last entry
   Verify: Read back the modified section to confirm formatting integrity

9. LOG REJECTIONS [OBSV-08]
   For each rejected candidate, append to rejections.jsonl:
     {"ts":"...","candidate":"...","reason":"...","confidence":"...",
      "source_agent":"...","source_run_ts":"..."}

10. UPDATE CURSOR
    Write .observer-cursor with new byte_offset (end of processed data),
    last_epoch_ms (from final processed event), last_run_id
    Crash safety: cursor updates AFTER all writes succeed for that run
```

**Tool Use:**

| Tool | Purpose in Observer | Configuration Notes |
|------|--------------------|--------------------|
| Read | Load obs.jsonl segments, read target memory files for dedup, read cursor file | Use `offset` param for large files; 2000-line limit per call is sufficient for single-run batches |
| Bash | `tail -c +N <file>` for byte-offset resume; `wc -c` to check file size vs cursor; line counting | Must handle Git Bash on Windows; paths with spaces need quoting |
| Grep | Search target files for existing entries matching candidate keywords | Used for dedup: find similar phrases in ## Pending Review and ## Permanent sections |
| Glob | Discover all write targets: `agent-memory/*/MEMORY.md`, `skills/*/insights.md` | Run once at start of invocation to enumerate valid targets |
| Edit | Append entries to ## Pending Review sections in memory files | Preferred over Write for surgical section modifications; preserves surrounding content |
| Write | Create/update .observer-cursor; append to rejections.jsonl | For rejections.jsonl: Read first, append new line, Write full content (or use Bash append) |

**State Management:**

| State | Location | Format | Lifecycle |
|-------|----------|--------|-----------|
| Read cursor | `.claude/logs/observations/<project>/.observer-cursor` | JSON: `{"byte_offset": 12345, "last_epoch_ms": 1713600000000, "last_run_id": "abc123"}` | Created on first run; updated at end of each invocation; reset if rotation detected |
| Rejection log | `.claude/logs/observations/<project>/rejections.jsonl` | JSONL: one rejected candidate per line | Append-only; same 10MB rotation + 30-day purge as obs.jsonl |
| Written entries | Target memory files (## Pending Review) | Markdown list items with confidence tags | Persist until /evolve promotes (→ ## Permanent) or reverts (git history) |

The observer is stateless between invocations except for the cursor file. No database, no in-memory cache, no session persistence. Each dispatch produces a fresh agent instance that reads cursor, processes new events, writes results, updates cursor.

**Context Window Strategy:**

Sonnet 4.6 as a subagent operates within a 200K token context window. Budget per invocation:

| Component | Estimated Tokens | Notes |
|-----------|-----------------|-------|
| System prompt (observer.md body) | ~3,000-4,000 | Includes few-shot examples, scope-test definitions, format specs |
| Injected skills (agent-protocols) | ~1,500 | SKILL.md content auto-injected |
| Auto-injected MEMORY.md | ~500-1,000 | Observer's own memory (should stay minimal) |
| Dispatch prompt from /evolve | ~300-500 | Project path + instructions |
| Events for one run | ~5,000-20,000 | Thinking blocks dominate: 10KB cap/turn * ~5 turns = ~50KB raw, ~12K-15K tokens |
| Target file reads (dedup checks) | ~2,000-5,000 | Grep results + Read of matching sections |
| Observer reasoning + output | ~3,000-5,000 | Chain-of-thought classification + formatted entries + rejections |
| **Total per run** | **~15,000-35,000** | Comfortable within 200K |

**Processing budget per dispatch:** 6-10 runs before context pressure builds. Prompt includes explicit stop condition: "If you have processed 8 runs and more remain, update the cursor and stop. Report remaining run count for the next invocation."

**Rotation fallback:** If `byte_offset > file_size`, the file has rotated. Fall back to `epoch_ms` comparison: scan from line 0 of the new obs.jsonl, skip events where `epoch_ms <= cursor.last_epoch_ms`. This is slower (full scan) but correct, and only triggers on rotation (rare: 10MB threshold).

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Adapted for Claude Code native subagent context: no Pydantic, no async runtime, no external SDK. All patterns are prompt-engineering and tool-use driven.

### 4b.1 Structured Outputs via Markdown Convention

Claude Code subagents have no Pydantic validation, no `response_format` JSON mode, no `instructor` library. Output structure is enforced entirely through prompt engineering, few-shot examples, and post-hoc validation using tool calls.

**Output schema (defined in observer system prompt):**

Entry for MEMORY.md / PLAYBOOK.md:
```
- [HIGH] researcher: Always verify Wikipedia dates against primary sources before committing to dossier (2026-04-18T10:22)
```

Entry for insights.md:
```
- [2026-04-20] [MED] Pin interpreter path in script invocations to prevent silent fallback to system Python (from: researcher, 2026-04-20T10:22)
```

Rejection JSONL:
```json
{"ts":"2026-04-20T14-30-00","candidate":"Always use WebFetch for paywalled sites","reason":"ambiguous-scope","confidence":"LOW","source_agent":"researcher","source_run_ts":"2026-04-20T10-22-00"}
```

**Enforcement strategy (no Pydantic equivalent):**

1. **Few-shot in system prompt** — 3 complete worked examples (one per memory tier) showing input events, reasoning chain, and final formatted output. The model pattern-matches these examples.

2. **Explicit format rules** — System prompt states: "Entry format is EXACTLY: `- [CONFIDENCE] agent-name: insight-text (evidence-pointer)`. No deviations. No extra fields. No multi-line entries."

3. **Self-validation via Read-back** — After each Edit to a memory file, the observer Reads back the modified section and checks: Does the new entry match the format? Is the confidence tag one of [HIGH], [MED], [LOW]? Is the agent name a valid agent from the project?

4. **Self-correction within invocation** — If read-back detects malformation (missing confidence tag, broken list formatting), the observer immediately issues another Edit to fix. One retry only. If still malformed after retry, log to rejections.jsonl with reason "format-error" and move on.

5. **File corruption guard** — Before writing to any target, observer confirms the target file contains a `## Pending Review` heading. If absent (file corrupt or unexpected structure), skip that target entirely and log with reason "target-file-corrupt". Never reconstruct file structure — that is a human/code-reviewer task.

### 4b.2 Synchronous-Only Execution

Claude Code subagents execute synchronously. There is no async runtime, no event loop, no streaming to parent, no parallel tool calls within an agent turn.

**Execution model:**
```
User runs /evolve
  → /evolve dispatches @observer (blocking call)
    → observer makes sequential tool calls: Read → Bash → Grep → Edit → ...
    → observer completes (or SubagentStop hook fires)
  → /evolve receives observer's final message
  → /evolve presents Pending Review entries to user
```

**The one mistake to avoid:** Attempting to speed up observer processing by dispatching it multiple times in parallel (e.g., one per project). Claude Code does not support concurrent subagent dispatches from the same conversation. The second `@observer` call will queue behind the first. Attempting workarounds with Bash background processes (`&`) spawning `claude` CLI calls would bypass the harness, lose tool sandboxing, and potentially double-count billing.

**Why sequential is correct for this use case:**
- The observer builds cumulative context within a run (event 5's meaning depends on events 1-4)
- Dedup checks require reading target files that may have been modified earlier in the same invocation
- Cursor updates must be ordered (process run N fully before starting run N+1)
- The latency budget (2-4 minutes per dispatch for 8 runs) is acceptable for a background maintenance task triggered by manual /evolve

### 4b.3 Prompt Engineering Discipline

**System prompt vs. dispatch prompt separation:**

| Component | Lives In | Content | Stability |
|-----------|----------|---------|-----------|
| System prompt | `observer.md` body (below YAML frontmatter) | Identity, processing pipeline, scope-test questions, entry format, few-shot examples, guardrails, stop conditions | Stable — changes only when classification logic or output format evolves |
| Dispatch prompt | /evolve command sends it at invocation time | Project slug, cursor file path, any per-run overrides (e.g., "only process researcher runs this time") | Dynamic — varies per invocation |

**Few-shot examples (inline, static):**

The observer's system prompt includes exactly 3 worked examples:

```markdown
### Example 1: Skill-layer insight (passes Q1 only)

Events: researcher run — tool_fail on Bash (crawl4ai script), then tool_post
on Bash with different interpreter path succeeding.
Thinking block: "The default python resolved to system Python which doesn't
have crawl4ai. Need to use the venv interpreter."

Candidate: "Pin interpreter path to C:/Users/iorda/venvs/crawl4ai/Scripts/python
in all crawl4ai script invocations"
Scope-test: Q1 YES (changes how crawl4ai-scraping skill runs), Q2 NO, Q3 NO
Target: .claude/skills/crawl4ai-scraping/insights.md
Confidence: [HIGH] — explicit failure + explicit fix in same run
Entry: - [2026-04-18] [HIGH] Pin interpreter to venv path — system Python
lacks crawl4ai deps, causes silent ImportError (from: researcher, 2026-04-18T10:15)
```

```markdown
### Example 2: Agent-layer insight (passes Q2 only)

Events: writer run — 3 consecutive Read calls to the same Research.md with
different offset values, then final Write of script.
```

**Token budget controls in prompt:**
- "Extract 0-3 learnings per run. NEVER more than 3."
- "Each insight is 1-2 sentences maximum. No multi-paragraph analysis."
- "If a run shows no actionable learnings, output: 'Run [agent_id] [timestamp]: 0 candidates extracted. Reason: [brief]' and move on."
- "After 8 runs processed, STOP regardless of remaining events. Update cursor. Report: 'Processed N runs, M remain for next invocation.'"

**Instruction priority (stated explicitly in prompt):**
1. Never process events where `agent_id === "observer"` (absolute rule)
2. Exactly one scope-test question must pass for acceptance (classification integrity)
3. Maximum 3 candidates per run (prevents over-extraction noise)
4. Evidence pointer is mandatory (traceability requirement)
5. Dedup before write (prevents redundancy)
6. Update cursor only after successful writes (crash safety)

### 4b.4 Context Window Management for Batch Log Processing

**The core challenge:** obs.jsonl is an unbounded append-only log that grows continuously between /evolve invocations. The observer cannot assume it will fit in context. Processing must be incremental and resumable.

**Strategy: Cursor-based windowed processing**

```
                 obs.jsonl (growing)
     ┌─────────────────────────────────────────────┐
     │ processed (before cursor) │ NEW (to process) │
     └──────────────────────────┬──────────────────┘
                                │
                        cursor byte_offset
                                │
                    ┌───────────┼───────────┐
                    │  Run A    │  Run B    │  Run C ...
                    │ (5 events)│(12 events)│
                    └───────────┴───────────┘
                    Process one run at a time
```

**Sizing estimates:**

| Scenario | Events | Estimated Size | Runs | Observer Invocations Needed |
|----------|--------|---------------|------|---------------------------|
| Light day (2-3 agent runs) | 50-150 | 100-300KB | 2-3 | 1 |
| Normal day (5-8 agent runs) | 200-500 | 500KB-1MB | 5-8 | 1 |
| Heavy day (15+ runs, researcher + writer + visual) | 500-2000 | 2-5MB | 15+ | 2-3 |
| Backlog (multi-day without /evolve) | 2000-8000 | 5-10MB | 30+ | 4-6 |

**Per-run context loading technique:**

1. Read cursor to get `byte_offset` and `last_epoch_ms`
2. Use `Bash: tail -c +<offset> obs.jsonl | head -c 100000` to load a chunk (100KB)
3. Parse JSONL lines; identify the first complete run (dispatch...complete bracket)
4. If the run's thinking blocks total >50KB of raw text, read only the first and last 3 turns (hypothesis: first turns show the problem setup, last turns show the resolution)
5. Process that run, write results
6. Advance cursor to byte position after that run's "complete" event
7. Repeat for next run (until 8-run or context-pressure limit)

**Context pressure detection (prompt instruction):**
"If at any point you notice your reasoning becoming less precise or you are unsure about event details you read earlier in this invocation, STOP processing immediately. Update cursor to the current position. Report: 'Context pressure detected after N runs. Stopping early.'"

**Rotation handling (10MB boundary):**
- obs.jsonl rotates when `pipeline-observe.js` detects size >= 10MB
- Old file moves to `obs.archive/obs-<timestamp>-<pid>.jsonl`
- New empty obs.jsonl created by next hook fire
- Observer detection: if `byte_offset > actual_file_size`, rotation happened
- Recovery: scan new obs.jsonl from line 0, skip events where `epoch_ms <= cursor.last_epoch_ms`
- Edge case: events with same `epoch_ms` that straddle the rotation boundary — the `last_run_id` field disambiguates (skip events from runs already completed)

### 4b.5 Cost and Latency Budget

**Cost model: Zero marginal cost per invocation**

All observer LLM calls are subagent dispatches covered by Claude Max subscription. No per-token billing, no API key, no usage tracking needed. The only "cost" is time (latency) and rate-limit consumption.

| Cost Factor | Impact | Mitigation |
|-------------|--------|-----------|
| Token billing | $0 (Claude Max flat rate) | N/A |
| Rate limit consumption | Each observer dispatch uses Sonnet quota for ~2-4 minutes | Run /evolve at most 2-3x per day; batch processing amortizes overhead |
| Opportunity cost | While observer runs, user cannot dispatch other agents in same conversation | Observer is fast (2-4 min); /evolve is intentional maintenance, not interrupt-driven |

**Latency budget per dispatch:**

| Phase | Expected Duration | Dominant Factor |
|-------|------------------|-----------------|
| Read cursor + initial obs.jsonl load | 2-5 seconds | Disk I/O + Bash tool overhead |
| Process one run (read events + extract + classify) | 8-20 seconds | LLM reasoning on thinking blocks; varies with run complexity |
| Dedup check per candidate (Grep + Read) | 3-6 seconds | Multiple tool calls; Grep is fast, Read adds overhead |
| Write one entry (Edit + verify Read-back) | 3-5 seconds | Two tool calls per entry |
| Cursor update | 1-2 seconds | Single Write call |
| **Total per run (with 1-2 entries written)** | **~20-40 seconds** | — |
| **Full dispatch (8 runs, typical day)** | **~3-5 minutes** | — |

**Acceptable because:**
- /evolve is a manual, infrequent operation (1-3x per day)
- User expects to review results, so waiting 3-5 minutes is not blocking
- No real-time UX depends on observer latency

**The cursor IS the cache:**
- No semantic cache, embedding store, or LRU needed
- Processed events are never re-processed (cursor advances monotonically)
- Dedup against target files is the only "repeated work" — and it must be fresh every time (files change between invocations)

**Sub-task model routing: Not applicable**
- No cheaper model available within Claude Code subagent dispatch
- All dispatches use the model specified in agent frontmatter (`sonnet`)
- If extraction volume becomes problematic, the lever is invocation frequency (run /evolve more often for smaller batches), not model routing
- Future optimization (if ever needed): split observer into two agents — a Haiku "scanner" that identifies interesting runs, and a Sonnet "extractor" that does deep analysis on flagged runs only. Not justified at current scale.

## 5. Evaluation Strategy

### Dimensions

| # | Dimension | Rubric | Measurement | Priority |
|---|-----------|--------|-------------|----------|
| 1 | Actionability | PASS: Insight prescribes a concrete behavioral change an agent can execute without interpretation — e.g., "Use `offset` param when Research.md exceeds 2000 lines" / FAIL: Insight is a truism, tautology, or vague observation — e.g., "errors should be handled" or "the writer agent writes scripts" | LLM Judge (calibrated against 10+ developer accept/reject decisions) | Critical |
| 2 | Evidence grounding | PASS: Every claim in the insight can be traced to a specific event sequence in the source obs.jsonl — agent name, tool call, and outcome are verifiable / FAIL: Insight describes a pattern not present in the source run data, or attributes behavior to the wrong agent, or cites a tool interaction that did not occur | Human (spot-check against source obs.jsonl during /evolve review) | Critical |
| 3 | Scope precision | PASS: Insight passes exactly one scope-test question, and the chosen target file is the one where consuming agents will benefit / FAIL: Insight passes 0 or 2+ questions (should be rejected but was written), OR passes 1 but the "consuming agent" test fails (a researcher tip in writer memory) | Code (validate exactly-one-pass rule) + Human (verify correct target on accept/reject) | High |
| 4 | Non-redundancy | PASS: Insight conveys knowledge not already expressed in the target file — either a new pattern, a meaningful refinement, or a superseding correction / FAIL: Insight restates existing knowledge with cosmetic differences — reviewer sees near-duplicate and rejects | Code (fuzzy string match against existing entries in target file) + LLM Judge (semantic similarity check) | Medium |
| 5 | Confidence calibration | PASS: HIGH applied only when evidence is unambiguous (error-recovery loop, explicit fix observed); MED for single-instance patterns; LOW for speculative inferences from thinking blocks only / FAIL: Uniform tagging (all HIGH or all LOW), or HIGH assigned to single-observation speculative patterns | Code (distribution check: flag if >70% of entries in a batch share same confidence level) + Human (calibration review during /evolve) | Medium |
| 6 | Self-loop prevention | PASS: Zero entries extracted from observer's own runs; all events with agent_id containing "observer" are skipped entirely / FAIL: Any entry references the observer's own behavior, or meta-observations about the observation process appear in Pending Review | Code (grep written entries and rejections.jsonl for agent_id "observer"; grep Pending Review for keywords "observer", "observing", "observation process") | Critical |
| 7 | Output format validity | PASS: Every written entry matches the exact format spec: `- [CONFIDENCE] agent: insight (timestamp)` for MEMORY.md/PLAYBOOK.md; `- [YYYY-MM-DD] [CONFIDENCE] insight (from: agent, timestamp)` for insights.md. Confidence is one of HIGH/MED/LOW. Agent name exists in project / FAIL: Malformed entries (missing confidence, wrong format, multi-line, invalid agent name) written to target files | Code (regex validation of all entries in Pending Review sections post-run) | High |
| 8 | Task completion | PASS: Observer processes all available runs up to the 8-run budget, updates cursor correctly, logs all rejections with valid reasons, and reports summary / FAIL: Observer crashes mid-run without updating cursor (causes re-processing), silently skips runs, or fails to log rejections | Code (verify cursor advanced; count written + rejected equals candidates extracted; verify rejections.jsonl has valid JSON per line) | High |

### Eval Tooling

**Override justification:** No external eval frameworks (Arize Phoenix, RAGAS, Promptfoo, etc.) are appropriate here. The observer runs inside Claude Code as a subagent — there is no HTTP endpoint to trace, no token billing to monitor, and no way to inject middleware. The existing smoke test pattern (Phase 1) provides the right model: a self-contained Node.js script with fixture data that validates observer behavior deterministically.

### Reference Dataset

**Size:** 15 examples minimum (to start), expand to 25 based on early /evolve sessions

**Composition:**

| Category | Count | Description |
|----------|-------|-------------|
| Clean extraction (happy path) | 5 | Runs with clear error-recovery or strategy patterns; expect 1-2 HIGH/MED entries each |
| Empty extraction (no-signal runs) | 3 | Mundane runs (simple file reads, no errors, no interesting decisions); expect 0 candidates |
| Scope ambiguity (rejection expected) | 3 | Runs where the candidate passes 2+ scope-test questions; expect rejection with "ambiguous-scope" reason |
| Self-loop bait | 2 | obs.jsonl containing observer's own events mixed with real agent events; expect observer events to be skipped entirely |

**Source material:** Real obs.jsonl excerpts from Phase 1 testing and early production runs. Each fixture includes:
- The obs.jsonl segment (5-20 events for a single run)
- The target memory files in their pre-observer state (for dedup testing)
- Expected output: which entries should be written (or rejected) and why

**Labeling:**
- Solo developer labels all examples manually during initial /evolve sessions
- Labels include: accept/reject decision, reasoning for the decision, which rubric dimension the decision tested
- Labels stored in `.claude/tests/fixtures/observer/README` as structured comments per fixture
- Re-labeling triggered when observer prompt changes significantly (scope-test rewording, confidence criteria change)

## 6. Guardrails

### Online (Prompt-Level, Per-Invocation)

These are enforced within the observer's system prompt — they run on every invocation as part of the agent's behavioral contract. No external middleware.

| Guardrail | Trigger Condition | Intervention |
|-----------|-------------------|--------------|
| Self-loop filter | Events where `agent_id` contains "observer" | **Block** — skip entirely, do not extract candidates. Hardcoded in prompt instruction priority #1. Violation = catastrophic (recursive noise). |
| Scope-test gate | Candidate passes 0 or 2+ scope-test questions | **Reject** — log to rejections.jsonl with reason "no-scope-match" or "ambiguous-scope". Never write ambiguous entries. |
| Max candidates per run | More than 3 candidates identified for a single run | **Cap** — take the 3 highest-confidence candidates, discard remainder. Log discarded count in observer's summary output. |
| Format self-check | Entry written does not match regex on read-back | **Retry once** — Edit to fix. If still malformed after retry, reject with reason "format-error". Never leave malformed entries in memory files. |
| Target file integrity | Target file missing `## Pending Review` heading | **Skip** — do not write to that file. Log with reason "target-file-corrupt". Prevents observer from corrupting file structure. |

### Offline (Flywheel)

These are tracked over time by the developer during /evolve sessions. No automated batch runner — the developer IS the batch reviewer.

| Metric | Sampling Strategy | Action on Degradation |
|--------|-------------------|----------------------|
| Accept/reject ratio | Every /evolve session: count accepts vs rejects across all proposed entries | If ratio drops below 50% acceptance over 3 consecutive sessions: review observer prompt for noise creep, check if scope-test questions need refinement |
| Confidence distribution | Every /evolve session: tally HIGH/MED/LOW across proposed entries | If >70% same tag: observer is not discriminating. Revise confidence criteria in system prompt. If all HIGH: evidence bar is too low. If all LOW: observer is being overly defensive. |
| Rejection reason distribution | Monthly: review rejections.jsonl grouped by reason code | If "ambiguous-scope" dominates (>50% of rejections): scope-test questions have boundary overlap — refine Q1/Q2/Q3 definitions. If "duplicate-of-existing" dominates: dedup is working but observer is not learning to avoid re-proposing — add "before extracting, check if this pattern category already has coverage" to prompt. |
| Diversity of extraction targets | Monthly: count unique target files written to | If >80% of entries go to a single file (e.g., always researcher MEMORY.md): curriculum collapse detected — observer only sees patterns from one agent type. Review whether obs.jsonl contains diverse agent runs, or whether prompt favors certain extraction patterns. |
| Stale insight detection | Quarterly: grep Pending Review and Permanent sections for insights referencing deprecated tools, old paths, or removed agents | If stale entries found: the observer is not the fix (it cannot detect staleness from obs.jsonl alone). Add a /prune step to /evolve that flags entries older than 60 days for re-evaluation. |

## 7. Production Monitoring

**Tracing Tool:** pipeline-observe.js (existing)

The observer's own tool events are captured to obs.jsonl by the same pipeline-observe.js hook that captures all agent activity. This means the observer's behavior IS traced — just not processed by itself (self-loop filter). The developer can manually inspect observer runs in obs.jsonl for debugging: search for events with `agent_id` containing "observer".

No external tracing platform (Langfuse, Arize Phoenix, LangSmith) is used. The system is local-only, single-user, and the trace data already exists in obs.jsonl. Adding a platform would provide visualization without adding signal.

**Key Metrics to Track:**

| Metric | How to Measure | Healthy Range |
|--------|---------------|---------------|
| Entries per invocation | Count lines written to Pending Review sections per /evolve run | 2-8 entries per invocation (for 5-8 runs processed). Consistently 0 = observer is too conservative. Consistently >15 = noise flooding. |
| Acceptance rate | `accepted / (accepted + rejected)` tracked manually during /evolve review | 50-80%. Below 50% sustained = prompt quality degrading. Above 90% = either perfect extraction or insufficient reviewer scrutiny. |
| Runs processed per invocation | Reported in observer's completion message | 5-8 typical. If consistently <3 despite available runs: context pressure or crash. If consistently hitting 8-run cap: consider running /evolve more frequently. |
| Cursor advancement | Compare .observer-cursor byte_offset before and after invocation | Must advance on every invocation that has events to process. Zero advancement with non-empty obs.jsonl = bug. |
| Rejection log growth | Line count in rejections.jsonl between invocations | Should be non-zero (proves filtering is active). If zero for 5+ invocations: observer may be accepting everything without discrimination. |

**Alert Thresholds:**

No automated alerting (no daemon, no monitoring service). Thresholds are mental models for the developer during /evolve review:

| Condition | Developer Response |
|-----------|-------------------|
| 0 entries proposed after processing 5+ runs | Check obs.jsonl quality — are events being captured correctly? Check observer prompt — is it too restrictive? |
| >3 entries rejected as "format-error" in one session | Observer prompt regression — format instructions or few-shot examples may have drifted. Inspect observer.md for recent edits. |
| Same insight proposed in 3 consecutive /evolve sessions | Dedup check is failing. Verify Grep is searching the correct sections. Check if insight is being written to Pending Review but not promoted (so it gets re-proposed each time). |
| Cursor file unchanged after observer dispatch completes | Observer crashed or exited early without updating cursor. Check obs.jsonl for observer events — look for the failure point. |
| All proposed entries target the same file | Curriculum collapse signal. Review obs.jsonl diversity — if input runs are diverse but output is homogeneous, the prompt favors one extraction pattern. |

**Smart Sampling Strategy:**

The observer's output volume is small enough (2-8 entries per /evolve session) that 100% human review is feasible and required. There is no sampling — every proposed entry is reviewed during /evolve.

For deeper investigation (when accept/reject ratio degrades):
1. Read the observer's own events in obs.jsonl to see its reasoning chain (thinking blocks captured by pipeline-observe.js)
2. Compare observer's reasoning against the source run data it was processing
3. Identify where reasoning diverged from expected classification

This "debug by reading the observer's trace" approach uses the same obs.jsonl infrastructure — no additional tooling needed.
